# Client Presentation Runbook (GitHub-first)

Этот runbook описывает, как проводить демонстрацию проекта клиенту без локального компьютера:
- основной путь: `GitHub Actions -> stage`;
- резервный путь: `GitHub Codespaces`.

## 1) Роли на демо

- Presenter: ведет продуктовый рассказ и показывает UI.
- Operator: мониторит health-check и логи.
- Backup operator: готовит fallback (Codespaces / скринкаст).

## 2) Тайминг подготовки

### За 3-5 дней

1. Закрыть изменения и убедиться, что `main` зеленый.
2. Прогнать quality gates:
   - backend: `npm run build && npm run test:smoke`
   - frontend: `npm run lint && npm run build`
3. Проверить, что `stage` environment и secrets заполнены.
4. Выполнить тестовый релиз в stage через workflow `Release Stage And Prod`.
5. Подготовить demo-аккаунты (passenger, driver, admin).

### За 1 день

1. Повторить dry-run релиза в stage:
   - `target_environment = stage`
   - `release_version = demo-<date>`
2. Проверить:
   - `GET /ops/health/live`
   - `GET /ops/health/ready`
3. Прогнать сквозной smoke:
   - создание заказа;
   - принятие заказа водителем;
   - смена статусов;
   - проверка админ-раздела.
4. Проверить fallback через Codespaces:
   - запустить `scripts/codespaces-demo.sh`;
   - убедиться, что открываются порты `3000` и `5173`.

### В день презентации

#### За 60 минут

1. Открыть Actions и убедиться, что последний stage deploy успешен.
2. Проверить health endpoint и stage URL в браузере.
3. Подготовить вкладки:
   - публичный лендинг;
   - пассажирский сценарий;
   - водительский сценарий;
   - админ-панель.

#### За 15 минут

1. Войти под нужными тестовыми аккаунтами.
2. Подготовить заказ-черновик для быстрой демонстрации.
3. Открыть runbook аварийного переключения (раздел 5).

## 3) Сценарий live-демо (15-20 минут)

1. Контекст и ценность (2-3 мин):
   - какие роли поддержаны;
   - почему фиксированные тарифы и realtime важны.
2. Пассажирский поток (4-5 мин):
   - создать заказ;
   - показать расчет и подтверждение.
3. Водительский поток (4-5 мин):
   - принять заказ;
   - обновить статус поездки.
4. Админ/операционный поток (4-5 мин):
   - открыть админ-панель;
   - показать оперативные данные/контроль.
5. Завершение (2 мин):
   - надежность (quality gates, health-checks, rollback);
   - следующий roadmap шаг.

## 4) Чеклист перед стартом звонка

- [ ] Stage URL доступен.
- [ ] `GET /ops/health/ready` возвращает 200.
- [ ] Demo-аккаунты работают.
- [ ] На GitHub есть успешный прогон `Release Stage And Prod`.
- [ ] Открыт fallback план (Codespaces).
- [ ] Есть записанный короткий скринкаст (на крайний случай).

## 5) Аварийный план (если stage упал)

### Вариант A: быстрый retry deploy

1. Запустить `Release Stage And Prod` повторно с тем же `release_version`.
2. Если не прошло, выполнить rollback через шаг workflow.
3. Перезапустить health-check.

### Вариант B: переход на Codespaces

1. Открыть `Code -> Codespaces -> Create/Resume`.
2. Запустить `bash scripts/codespaces-demo.sh`.
3. Открыть forwarded port `5173` и продолжить показ.

### Вариант C: fallback на запись

1. Показать заранее записанный сценарий.
2. Параллельно восстанавливать stage в фоне.

## 6) Действия после презентации

1. Зафиксировать вопросы клиента и возражения.
2. Создать follow-up: ссылки на репозиторий, demo URL, roadmap.
3. Сохранить incident notes, если были технические сбои.
4. Обновить backlog и сроки следующего демо.
